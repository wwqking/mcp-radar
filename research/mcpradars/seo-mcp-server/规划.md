# mcp server 关键词 → 页面规划（保证能被谷歌收录）

> 数据源：`mcp-server_broad_us_2026-07-24.csv`（348 词，US，KD 0-29）
> 日期：2026-07-24　｜　站点：mcpradars（MCP 目录站）

## 一句话结论

**不能「一词一页」照单全收。** 348 个词里 264 个是「{某工具} mcp server」品牌词，
其中 **160 个工具你库里没有真实 server 数据**——给它们建页就是空壳页（thin content），
谷歌会判 doorway pages 并**降权整站**。真正能安全收录的路径是下面 4 层。

## 收录的底层逻辑（为什么这样分）

谷歌收录看的是**每页有没有独特、有用的内容**。据此：

- 有**真实 server 数据**（安装命令 / GitHub stars / TrustScore / 能力卡）→ 独特 → ✅ 收录
- 有**原创教程长文** → 独特 → ✅ 收录
- 只是套模板换个工具名、背后无数据 → 空壳 → ❌ 被罚

## 落地顺序建议

1. **先扩第 1 层的 Wave 1（12 个落地页）** —— 有现成机制+真数据，最快见效、零收录风险。
2. **同时补第 3/4 层里已有页的量**（claude-code-config、支柱页），再写 2-3 篇高价值新教程。
3. **把第 2 层 160 个喂采集器**，跑几轮把热门工具的 server 补进库。
4. 采集补到的工具，滚动升级成第 1 层落地页。

## 反面提醒（避免被罚）

- ❌ 不要给 160 个缺 server 的工具建「暂未收录，推荐相似」占位页——那是典型 doorway page。
- ❌ 不要把 39 个教程词各建一页——同义词内容雷同会自相蚕食（cannibalization）。
- ❌ 不要为 logo/icon 建内容页。
- ✅ 每个落地页确认背后 server 是主流实现，不是低质同名 fork。
